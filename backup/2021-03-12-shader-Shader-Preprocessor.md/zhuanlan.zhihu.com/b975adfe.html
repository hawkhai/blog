自制游戏引擎 - 5 Shader Preprocessor

Tanki Zhang
Making Shiny Pixels ✨

最近正在全面把 Engine 从 OpenGL 1.x 的笨笨的 Pipeline 迁移成 Programmable Rendering Pipeline。简而言之就是支持 Shader 了。一系列工作有机会再说，今天主要谈下给 Shader 加的两个 Proprocessor。本篇主要是 Compiler Topic，如果不感兴趣可能现在就可以溜了【

简而言之，我希望我的 Preprocessor 要处理两大问题：

  1. 支持 Engine 能够自定义插入 #define xxx。这样一来 Engine 可以有一系列可配置项，并根据用户的选择，在提交 Shader file 到 GPU 之前，插入一些 Define Directives，做到一些流程控制。
  2. GLSL 并不支持 #include，但事实上 include 其他文件，比如自己的函数库之类的是一件被各大Engine普遍支持的功能（对，然而 OpenGL 到现在为止(450)还都基本不支持）。

如果需要比较好的实现这两个功能，需要小心以下几个问题：

  1. 在插入自定义内容以后，行号会变，报错的地方会乱。所以需要在插入内容后，添加 #line 来为 Compiler 提供 hint。
  2. #define 需要插入在实际的 #verison 之后。划重点，实际的 #version。这意味着混在注释里的是不可以的。
  3. 同 2，只需要 include 真的#include，注释里的需要忽略。
  4. Include 的文件里很可能还有 Include，所以需要继续 parse。这个时候，请再次小心第 1 点。
  5. Include 需要小心 Circular Dependencies
  6. Include 要维护 Include Directory 保证各种 Include 文件的路径都是合法的。（清完全参照C/C++能支持的方式）


#version & #define

要准确找出 #version，其实我们需要 parse 的是这个（偷懒不写 EBNF 了，就用一个类似于正则的方式来表达下）：

[Whitespace | Comment |InlineComment]*#[Whitespace]version[Whitespace | Comment | InlineComment]*[Whitespace][Digits][Whitespace]

解释一下，[Whitespace | Comment | InlineComment]* 表示 0 或多个 whitespace 或者 Comment 或者 InlineComment。

如果只是为了插入 #define 的话事实上只需要 parse 到 #version 就好了，但是为了之后的内容，我这里继续 parse 拿到了 version。

Whitespace = ' ' | '\t'

Comment = "//"…'\n'

InlineComment = "/*"…"*/"

这里我们需要用 LL(1) 来 parse，每次向前可能需要多读一个符号。对于#version来说，最好的一点是因为version需要出现在正式内容的最开始，所以之前只有空格或者注释，所以比较容易parse。大致逻辑如下：

bool consumeVersion (const char*& c, uint& version) {
  constexpr size_t VERSION_KEY_LEN = 7;
  passWhitespaceAndComment(c);

  // #
  if (*c != '#') return false;
  c++;

  consumeWhitespace(c);

  // version
  if (strncmp(c, "version", VERSION_KEY_LEN)) return false;
  c += VERSION_KEY_LEN;

  consumeWhitespace(c);

  if (!isdigit(*c)) return false;

  version = (uint)std::strtoul(c, nullptr, 0);
  while (!isReturn(c)) {
    c++;
  };
  return true;
}

#include

逻辑和 #version 类似，但是事实上这个问题如果要完全处理，我们需要把整个文件全 parse 一遍，因为用户的 Include 可能出现在文件的任何一行。这显然工作量会大很多，所以事实上，在我的实现中，暂时只逐行寻找 Include directive，但是并没有处理所有 Include 被 inline comment(/*…*/) 包围的情况（比如comment start 在第一行，Comment end 在第三行）。这些 side case 暂时只要知道没办法 handle 就可以了，毕竟基本不是正常的使用场景，属于非常 side 的 case。

定义一个 function:

bool ShaderStage::parseBody(
    const Path& currentFile, std::string& body, 
    uint currentLine, std::set<Path>& includedFiles)

主文件剩下部分和被include的文件都需要靠这个 function 来处理，注意到
includeFiles 是一个 set，每当我们即将 include 一个新的文件时，就需要将该文件名加入 set 中，这样能保证我们不重复
include 相同的文件，并且可以避免 Circular
Dependencies。 currentLine 是当前的行号，用来插入额外的 #line 信息。


#line

关于 #line，直接贴代码：

std::string ShaderStage::makelineDirective(uint version, uint line, const Path& filename) {
  // glsl 330 and up, #line means next line, before means current line
  if(version >= 330) {
    line++;
  }
  return Stringf("#line %u \"%s\"\n", line, filename.filename().string().c_str());
}

这就是之前为什么需要把 version 拿出来，不一样的 version 对于 line 的处理有些微妙的差别…


Nonexcept & std::move

然后最后来讲一个 C++ 语言层面的内容。应该很多人都知道，要尽量将 move constructor 申明为 nonexcept。因为如果它是 nonexcept 的话，可以对它进行很多的优化，可是具体有些什么优化呢？这里随便举个栗子。

std::vector 的 push_back 如果触发了 re-allocate，如果对象的move constructor 是 nonexcept 的话，vector 会直接调用 move constructor，而不是 copy constructor。push_back作为一个语言层面的原子操作，只能成功，或者失败。如果 move 会触发 exception，那就意味着有失败的可能，如果失败了，就需要回滚。如果已经 move 了，就没办法回滚。所以只能调用 copy constructor。总的来说因为 nonexcept 提供了 strong exception guarantee，而如果没有 nonexcept，就只能提供 basic guarantee。（side note, std::vector 的 push_back 使用 std::move_if_nonexcept 来实现这层逻辑）

还有一点要注意的是，nonexcept 是 api 的一部分，和我们加的各种 const 是一样的（并且事实上在 C++ 17 中，nonexcept 已经成为类型的一部分了），是在向使用者提供 promise。所以，虽然说要尽量将能申明 nonexcept 的东西申明为 nonexcept，但是不能因为现在的实现允许申明为 nonexcept 就申明为 nonexcept。总而言之，要是真的肯定 nonexcept 才申明成 nonexcept，就好像我们在 member function 最后加的 const 一样。
